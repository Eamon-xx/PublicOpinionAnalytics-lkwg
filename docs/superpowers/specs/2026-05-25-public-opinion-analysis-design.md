# 舆情分析通用工具设计

## 背景

当前输入是一类固定结构的评论 CSV，后续总样本量预计可达百万级。评论来源为某视频平台评论区，包含评论文本、层级关系、时间、互动数据和基础用户画像字段。样本通常按热度抓取，存在明显的头部评论、模板化短回复、刷屏式动员文本，以及评论正文中包含换行的情况。

本设计目标不是针对单个样本做一次性分析，而是建立一套可复用的离线分析工具链，后续只要输入同结构数据即可重复执行。

## 目标

工具需要保证以下结果：

1. 原始 CSV 能被稳定解析，不因评论换行或字段格式差异导致错行。
2. 清洗后产出统一结构的评论事实表，支持百万级离线分析。
3. 低信息评论与有效观点评论被分流处理，避免污染主题和情感结果。
4. 大模型只负责理解高价值文本，不承担适合规则处理的噪声分类。
5. 模型标签、互动指标和用户画像能汇总到同一分析视图中。
6. 后续能基于同一套代码处理新的同结构评论数据。

## 约束与风险

- 原始数据可能不是全量抓取，不能把评论数直接当作真实总体规模。
- 评论区存在大量模板化文本，若直接做情感分析会高估某些情绪或立场。
- 百万级数据不适合全部直接调用大模型，成本和耗时都会失控。
- 用户画像字段存在天然偏差，例如性别保密、IP 属地非真实常住地、会员信息不代表真实消费能力。
- 同一用户可能多次发言，同一模板可能被多人复读，因此需要区分“评论数”和“用户数”。

## 方案选项

### 方案 A：全量评论直接送大模型

优点：流程直观，开发最省事。

缺点：成本高，噪声多，模型会被“顶”“冲”等低信息文本拖累，结果不稳定。

### 方案 B：规则预清洗 + 分层送模 + 聚合分析

优点：成本可控，结果可解释，适合百万级数据，可复用性强。

缺点：前期需要定义清洗规则、标签体系和中间数据结构。

### 方案 C：先聚类或向量检索，再做抽样标注

优点：能进一步压缩调用量，适合长期迭代。

缺点：实现更复杂，对首版落地不够直接。

## 推荐方案

采用方案 B，后续保留向方案 C 演进的空间。

原因：

1. 规则清洗可以稳定处理换行、空白、数值类型和重复数据。
2. 低信息文本可以规则化识别，不值得消耗模型调用。
3. 高价值文本再送模型，可以把成本集中在真正有分析意义的内容上。
4. 最终结果可以直接用聚合统计完成，不需要把“综合分析”再交给模型二次判断。

## 工具边界

本工具链包含以下模块：

1. 数据读取与标准化
2. 文本清洗与噪声识别
3. 评论价值分层
4. 模型输入导出
5. 模型结果回填
6. 用户画像与标签聚合分析
7. 报表与样本导出

本工具链暂不包含：

1. 在线实时分析
2. 自动爬虫采集
3. 前端可视化系统

## 数据流设计

### 第 1 层：原始层

输入文件：`remark.csv`

保存原则：

- 原始文件只读保存，不做覆盖修改。
- 每次分析都生成新的中间层和结果层文件。

### 第 2 层：标准化层

输出建议：`data/normalized/comments.parquet`

字段示例：

- `comment_id`
- `parent_comment_id`
- `user_id`
- `username`
- `user_level`
- `gender`
- `vip_flag`
- `signature`
- `ip_location`
- `comment_text_raw`
- `comment_text_clean`
- `comment_time`
- `reply_count`
- `like_count`
- `avatar_url`
- `is_root`
- `reply_depth`
- `is_direct_reply_to_root`
- `source_file`

这一层负责：

- 标准 CSV 解析
- 数值和时间字段类型转换
- 统一字段命名
- 基础去重
- 结构字段补充

### 第 3 层：规则标签层

输出建议：`data/processed/comments_rules.parquet`

新增字段示例：

- `char_len`
- `token_len`
- `is_empty_text`
- `is_short_text`
- `is_only_emoji_or_punct`
- `is_template_text`
- `template_group`
- `is_low_info`
- `is_mobilization`
- `rule_topic_tags`
- `rule_risk_tags`
- `analysis_priority`

这一层负责：

- 识别“顶”“冲”“1”“支持”等低信息文本
- 识别模板化复读文本
- 识别明显动员型文本
- 给每条评论一个分析优先级

## 评论分层规则

建议把评论分成 4 类：

### A 类：低信息噪声

示例：`顶`、`冲`、`1`、纯表情、纯标点。

处理方式：

- 不送模型做细粒度情感分析
- 保留在动员强度、刷屏强度统计中

### B 类：短文本但有明确立场

示例：`不接受`、`必须处罚`、`还没认错呢`。

处理方式：

- 可送模型，或先规则打初标签再由模型校正

### C 类：长文本或高互动文本

示例：长评论、高赞评论、被大量回复评论。

处理方式：

- 优先送模型
- 作为主题提炼和观点归因的核心样本

### D 类：模板复读文本

示例：同一短句被大量重复发送。

处理方式：

- 对模板建立 `template_group`
- 可以对模板代表样本送模，其余评论继承模板标签

## 模型标注设计

模型输入建议使用 JSONL，而不是把全量主数据改造成 JSON。JSONL 只用于批量推理请求。

单条模型输入字段建议：

- `comment_id`
- `comment_text_clean`
- `parent_comment_id`
- `is_root`
- `like_count`
- `reply_count`
- `char_len`
- `rule_flags`

模型输出字段建议：

- `sentiment`
- `stance`
- `topic_tags`
- `emotion_intensity`
- `risk_tags`
- `is_low_info_confirmed`
- `is_mobilization_confirmed`
- `summary_reason`

### 标签建议

`sentiment`

- 正面
- 中性
- 负面
- 混合
- 无法判断

`stance`

- 支持官方处理
- 反对官方处理
- 要求进一步处理
- 产品诉求导向
- 跟随主评动员
- 观望
- 无法判断

`topic_tags`

- 认错态度
- 处理方案
- 处罚诉求
- 产品改动
- 自定义功能
- 性别或群体对立
- 公关质疑
- 无明确主题

`risk_tags`

- 辱骂攻击
- 煽动动员
- 群体对立
- 极端表达
- 无明显风险

## 模型调用策略

推荐顺序：

1. 规则先筛掉明显低信息噪声。
2. 对高价值评论直接送模型。
3. 对模板复读文本只送模板代表样本。
4. 对剩余边界样本做抽样复核。

这样优化的是成本和稳定性，代价是需要维护一套规则和模板归并逻辑。

## 用户画像融合设计

用户画像不单独送模型二次理解，而是作为聚合维度参与统计分析。

基础分析口径：

- 评论数
- 去重用户数
- 点赞加权评论数
- 模板去重后的评论数

核心交叉维度：

- `sentiment x user_level`
- `stance x vip_flag`
- `topic_tags x ip_location`
- `risk_tags x gender`
- `topic_tags x comment_time`
- `is_mobilization x ip_location`
- `stance x like_count_bucket`

为避免被刷屏误导，所有结果至少同时输出两种口径：

1. 评论数口径
2. 去重用户数口径

若业务侧关注舆情影响力，再补充：

3. 点赞加权口径

## 结果产出

首版工具建议输出：

1. 清洗后的主表
2. 待送模 JSONL
3. 模型结果回填后的最终主表
4. 标签分布统计表
5. 用户画像交叉统计表
6. 高频模板文本表
7. 高风险评论样本表
8. 高价值评论样本表

## 测试重点

至少覆盖以下验证点：

1. 含换行评论能否正确解析。
2. 数值和时间字段能否稳定转换。
3. 去重规则是否正确。
4. `顶/冲/1` 等低信息文本能否被正确识别。
5. 模板归并逻辑是否稳定。
6. 模型结果回填后能否按 `comment_id` 正确关联。
7. 聚合统计是否同时支持评论数和去重用户数。

## 首版实现范围

首版只做离线批处理，优先完成：

1. CSV 标准化
2. 规则打标
3. JSONL 导出
4. 模型结果合并
5. 聚合统计导出

暂不实现复杂聚类、向量检索或可视化页面。
